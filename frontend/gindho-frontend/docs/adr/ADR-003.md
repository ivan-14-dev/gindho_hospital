# Architecture Decision Records (ADRs)

## ADR-001: Architecture Frontend Enterprise

### Statut
Accepté

### Contexte
Le frontend GinDHO doit être maintenable, testable et extensible pour une application hospitalière.

### Décision
Adopter une architecture feature-based avec séparation claire des responsabilités:
- `app/` : Initialisation et configuration globale
- `shared/` : Composants et utilitaires réutilisables
- `entities/` : Types métier et entités
- `features/` : Workflows et logique métier
- `services/` : Accès API
- `pages/` : Pages de l'application

### Conséquences
- Meilleure maintenabilité
- Tests plus faciles
- Scaling d'équipe

## ADR-002: TanStack Query vs Redux

### Statut
Accepté

### Contexte
Gestion du state serveur pour les données patients, rendez-vous, etc.

### Décision
Utiliser TanStack Query plutôt que Redux:
- Caching intelligent
- Revalidation automatique
- Moins de boilerplate

### Conséquences
- Bundle plus léger
- API plus simple pour les mutations
- Moins de tests à écrire

## ADR-003: RBAC Frontend

### Statut
Accepté

### Contexte
Contrôle d'accès aux modules hospitaliers.

### Décision
- Permissions côté API uniquement (source de vérité)
- Filtrage UI par rôles/permissions
- Routes protégées côté client ET serveur

### Conséquences
- Double sécurité
- UX améliorée (pas d'erreurs 403 inattendues)
